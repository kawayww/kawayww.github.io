我们很高兴宣布 GNU Guix 1.4.0 的发布!

本次发行包含 ISO-9660 安装镜像, 虚拟机镜像, 以及可在现有 GNU/Linux 发行版上安装 Guix 包管理器的 tar 包, 无论是从源码还是从二进制出发都可以, 详情可查看 下载页面
Guix 用户可以通过 guix pull 完成更新

距离 上个版本 已过去 18 个月
这既说明作为 rolling release, 用户其实一直在通过 guix pull 持续获取新功能, 也坦率地反映了我们在发布流程上还有继续改进的空间
在这段时间里, Guix 收到了约 29000 次提交, 来自 453 位贡献者, 项目还完成了 维护者轮换, 以 teams 的形式重新组织协作, 并庆祝了 十周年

这篇文章会概括本次发行中最值得关注的变化, 也是 Guix 历史上可能最长的一版 release notes, 所以请坐稳, 放松, 慢慢看

> 改进的软件环境管理

Guix 在软件环境管理方面一直非常出色
最初为此设计的 guix environment 命令界面略显笨拙, 现在它已经被 guix shell 取代, 尽管我们承诺至少会把旧命令保留到 2023 年 5 月 1 日
guix shell 不仅对开发者有用, 也适合那些想临时试用某个软件, 但又不想把它正式加入 profile 的用户
例如, 如果你想玩 SuperTuxKart, 又不想把它装进工作电脑的 profile, 那么可以直接运行:

guix shell supertuxkart -- supertuxkart

除了界面更简洁, guix shell 还通过缓存显著提升了性能
它现在会自动识别目录中的 guix.scm 与 manifest.scm, 这样其他开发者只需进入项目目录并运行 guix shell 就能立刻开始工作, 无需再额外传参
另外, --export-manifest 也能帮你把命令行参数 “转换” 成 manifest, 更适合后续纳入版本控制
更多信息请参见 手册

guix shell 的另一项重要创新是可选的 FHS 模拟, 也就是 filesystem hierarchy standard
Guix 的 store 天生并不遵循 FHS, 这会让部分默认假定 /bin, /lib 等目录存在的软件无法直接运行
为此, guix shell 新增了 --emulate-fhs (-F) 选项, 与 --container (-C) 结合后, 能创建一个遵循 FHS 的容器环境
例如:

$ guix shell -CF coreutils -- /bin/ls -1p /
bin/
dev/
etc/
gnu/
home/
lib/
lib64
proc/
sbin/
sys/
tmp/
usr/

另一个重量级新特性是 Guix Home
简而言之, Home 把 Guix System 的 declarative 思路扩展到了你的 home environment: dot files, 服务, 软件包 都可以统一声明, 然后在真实的 $HOME 或容器里 实例化该环境
如果你已经在用版本控制管理 dot files, 或你不想在下次换电脑时再次花几天几周重配环境, 那这就是你需要的工具
你可以先看看 David Wilson 在十周年活动上的 介绍视频, 然后再去读 Guix Home 手册

package transformation options 也继续增强
新的 --tune 选项允许你针对特定 CPU micro-architecture 调优软件包, 从而使用更新的 SIMD 指令集, 例如最近 AMD/Intel CPU 上的 AVX-512
对某些工作负载, 尤其是线性代数计算, 这会带来相当可观的提升, 细节可见 这篇文章

从 1.3.0 以来, 项目还开始维护 另一座 build farm, 地址为 https://bordeaux.guix.gnu.org
它与 ci.guix.gnu.org 完全独立, 带来了两点好处: 一是可以通过 guix challenge 对不同农场产出的 substitutes 进行交叉验证, 二是在其中一座农场下线时提供冗余
现在 Guix 默认就会从这两座 build farm 中获取 substitutes
此外, 我们也修复了 一个 bug, 让 Guix 在某个 substitute provider 宕机时可以更平滑地切换到另一个

如果你已经在享受把整批机器 declarative 地部署出去的乐趣, 那你大概率也会喜欢 guix deploy 新增的 --execute 选项

> 更强的发行版

发行版本身也出现了大量变化
首先, Guix System 安装器收到了许多 bug fix, 还加入了新的 crash 调试信息上报机制, 这会帮助开发者更快定位那些只在少见配置上出现的问题

应用启动速度也获得了明显改善
我们引入了 按应用划分的 dynamic linker cache, 大幅减少了共享库查找过程中 stat 与 open 的调用次数, 这项工作甚至还启发了其他生态中的类似尝试

Guix System 现在使用 GNU Shepherd 0.9, 它修复了不少短板, 改进了 logging, 并加入了 systemd 风格的 service activation 与 inetd 风格的服务启动方式
与此同时, guix system edit 作为新的子命令加入进来, 让用户可以和 guix system search, guix system extension-graph 一起, 以更多角度查看系统服务

系统现在还新增了 15 个服务, 其中包括 Jami, Samba, fail2ban, 以及 Gitile

操作系统配置中还加入了新的 swap space 声明接口, 它比过去的方案表达能力更强也更灵活

同样地, 静态网络配置 接口也被整体重写
在 GNU/Linux 上, 你现在几乎可以 declarative 地完成 ip 命令能做的那些事, 同时还能得到静态检查, 从而避免把明显有问题的配置部署出去

本次发行新增了 5300 多个软件包, 总数已逼近 22000, 按 Repology 的统计, Guix 已进入规模最大的发行版前十
众多软件更新中, 最值得一提的是 GNOME 42 已正式可用
KDE 还未完全到位, 但几十个 KDE 包已经加入进来, Qt 6 也已经可用
发行版还自带 GCC 12.2.0, GNU libc 2.33, Xfce 4.16, Linux-libre 6.0.10, LibreOffice 7.4.3.2, 以及支持 JIT 编译的 Emacs 28.2

另外, 由于 Python 2 已在 2020 年正式走到 “end of life”, 我们这次删除了 500 多个以 python2- 开头的软件包, 其中包括 python2-numpy 与 python2-scipy 这类大包
如果你仍然依赖它们, 现在有两种办法: 一是使用 guix time-machine 跳回仍包含这些包的旧提交, 二是借助 Guix-Past channel 在今天的环境中重新构建这批旧软件
这对 scientific computing 场景尤其有帮助

除此之外, 我们的网站终于也拥有了 新的 package browser
它提供了稳定的软件包 URL, 例如 https://packages.guix.gnu.org/packages/PACKAGE

更多值得关注的变化与 bug fix, 请查看 NEWS

> 支持长期可复现性

Guix 的一个核心特性是对 reproducible software deployment 的支持
其中很重要的一点, 是能够从 Software Heritage archive 中找回源码
过去 Guix 已能在软件包源码消失时回退到 Software Heritage, 而从 1.4.0 起, 如果某个 Guix channel 的原始 URL 不再可达, 它的源码也会自动从 Software Heritage 获取

除此之外, Guix 现在也具备了恢复 tar.gz 这类源码 tarball 的能力
Software Heritage 保存的是 tarball 的 内容, 但并不直接保存 tarball 文件本身
而 Guix 中大多数 package definition 都是围绕 tarball 编写的, 并且要求校验 tarball 本身 的内容哈希
为填补这道缝隙, Timothy Sample 开发了 Disarchive, 它既能提取 tarball 的元数据, 也能用已提取的元数据与实际文件重新组装出原始 tarball

Guix 项目还搭建了 持续集成任务, 用于构建可在 disarchive.gnu.org 获取的 Disarchive 数据库
该数据库包含了所有软件包引用 tarball 的元数据
当源码 tarball 消失后, Guix 现在会透明地从 Disarchive 获取元数据, 再从 Software Heritage 取回文件内容, 最终重建原始 tarball
按 2022 年 1 月发布的 “Preservation of Guix Report” 统计, 软件包引用的 .tar.gz 文件中, 已有近 75% 能通过 Disarchive 与 Software Heritage 完整归档
运行 guix lint -c archival PKG 还能直接查看 PKG 的归档状态
更多细节, 可参见 Guix-HPC 年报

这是向长期可重现部署迈出的重要一步, 因为它第一次让我们有了在保留源码与二进制关联关系的前提下, 重新部署过去软件环境的工具

> 应用打包与系统镜像

guix pack 这个用于创建 application bundle 的命令也扩展了很多能力
例如 guix pack -f deb 现在可以直接产出可在 Debian 及其衍生发行版上安装的独立 .deb 包, 新增的 --symlink 选项则会在镜像内部创建符号链接

在系统层面, 新的 guix system image 命令 统一取代了过去分散的多个 guix system 子命令
它为各种镜像类型提供了单一入口: 原始磁盘镜像, QCOW2 虚拟机镜像, ISO9660 光盘镜像, Docker 镜像, 甚至 适用于 WSL2 的镜像
这套命令还配备了 更高层的接口, 可 declarative 地描述存储格式, 分区布局与操作系统内容
为了降低使用门槛, 系统内置了一批 image type:

$ guix system image --list-image-types
The available image types are:
  - rock64-raw
  - pinebook-pro-raw
  - pine64-raw
  - novena-raw
  - hurd-qcow2
  - hurd-raw
  - raw-with-offset
  - iso9660
  - efi32-raw
  - wsl2
  - uncompressed-iso9660
  - efi-raw
  - docker
  - qcow2
  - tarball

其中既有面向 Pine64 设备的镜像类型, 也有面向 GNU/Hurd 的镜像类型
如果你想生成可供 QEMU 使用的 QCOW2 虚拟机镜像, 现在只需:

guix system image -t qcow2 my-operating-system.scm

如果你想在 x86_64 机器上为 Pine64 板子生成镜像, 只需:

guix system image -t pine64-raw my-operating-system.scm

pine64-raw 会自动把软件交叉编译到 aarch64-linux-gnu, 并配上对应的 U-Boot 变体 作为引导器, 想象空间非常大

> 更舒适的打包体验

对 packager 来说, 1.3.0 之后最显眼的变化之一是 package simplification
最直观的效果是 package definition 看起来清爽了很多:

(package
  ;; ...
  (inputs (list pkg-config guile-3.0)))

而不再是过去那种带 “input label” 的古典写法:

(package
  ;; ...
  (inputs `(("pkg-config" ,pkg-config)
            ("guile" ,guile-3.0))))

新的 guix style 命令能自动把旧式写法转换成新式写法, 也能按当前推荐风格格式化整个 Scheme 文件, 对刚接触这门语言的人尤其友好

这还只是冰山一角
新的 modify-inputs 宏让 package input 的操作更简单也更清晰, G-expression 也可以更自然地用于 package phase 之中
此外, 新引入的 field sanitizer 机制可以在几乎不增加常见运行时开销的前提下, 校验部分字段的合法性, 比如 license 字段的类型, 以及 description 与 synopsis 中的 Texinfo 语法
我们希望这些改动能显著降低打包的入门难度

guix build 现在还新增了 --list-systems 与 --list-targets, 用于列出受支持的 system type 与 cross-compilation target triplet
在底层, 新的 (guix platform) 模块允许开发者以更抽象的方式定义 “platform”, 也就是 CPU 架构与操作系统的组合, 把之前散落各处的信息重新统一起来

除此之外, packager 也可以把某些适合做 CPU micro-architecture 优化的软件包标记为 “tunable”, 然后通过 --tune 启用这类优化

Python 打包同样经历了重要调整
首先, python 包现在会读取 GUIX_PYTHONPATH, 而不是传统的 PYTHONPATH, 这能避免 Python 不小心拾取到 Guix 之外提供的包
其次, 新的 pyproject-build-system 实现了 PEP 517, 与现有的 python-build-system 形成互补, 未来两者还有可能进一步合并

而在 “免费获得更多打包支持” 这件事上, 也有新惊喜
guix import 新增了对 minetest, elm, egg, hexpm 等上游仓库的支持, 既覆盖游戏扩展, 也覆盖 Erlang/Elixir 包生态
既有 importer 也持续得到改进
同时, guix refresh 现在还多了一个 generic-git updater

> 试试看

如果你想开始使用 Guix, 现在大致有几种路径:

  • 安装脚本, 可让你把 Guix 快速安装到 现有 GNU/Linux 发行版 之上
  • Guix System 虚拟机镜像, 可配合 QEMU 使用, 不碰现有系统就能快速体验
  • 直接 把 Guix System 作为独立发行版安装, 安装器会引导你完成初始配置

想了解所有可用的安装选项, 请查阅 下载页面, 也欢迎通过 联系页面 与我们交流

Enjoy!
