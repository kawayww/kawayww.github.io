Guix 带有一套名为 grafts 的机制, 让我们即便面对依赖图深处的 core package, 也能及时向用户提供 security updates
大多数用户都认可 grafts 带来的好处, 但与此同时, 初次接触 Guix 的人往往也会立刻碰到 graft 实现给用户体验带来的副作用
这个问题其实已经存在了很长一段时间, 直到 1.1.0 才算真正把它解决掉

这篇文章会回顾 grafts 的实现方式, 它带来了哪些问题, 以及我们最终是怎么处理这些问题的
内容会比较深入, 主要围绕 Guix 的核心机制展开, 希望它既能给一般读者带来一些启发, 也能让偏爱 functional programming 的朋友觉得有趣

> 这个 “graft” 到底是什么

grafts 在 Guix 发展的早期就已经引入了, 主要是为了解决 functional software deployment 在实践里最显著的一个短板
简单来说, Nix 和 Guix 所采用的 functional deployment 意味着, 只要某个包发生变化, 所有依赖它的东西就都必须重新构建, 或者重新下载
因此, 如果你要为 C library 或 Bash 部署一个 security fix, 原则上就得把整条链路上的东西都重建一遍
即便背后有一座很大的 build farm, 这仍然会显著拖慢修复的交付速度, 用户不是得在本地自己重建, 就至少得把一大堆二进制重新下载一遍

为了解决这个问题, Guix 的开发者可以在 package definition 里指定一个 replacement
如果我们手上有一个 libc 的 bug fix, 开发者通常会做两件事: 一是定义出修复后的 libc 包, 二是在原始 libc 的 package definition 里加上 replacement 字段, 让它指向这个修复版本
这样一来, 真正需要构建的只有修复后的 libc 本身
在构建其他包时, 这个修复版 libc 会自动 graft 到目标包上, 于是得到的包最终引用的就是修复后的 libc
更完整的说明可直接参考 手册

当 Guix 把高层的 package definition “lower” 成底层的 derivation 时, 它会遍历整个依赖图, 找出一组 potentially applicable grafts
这里之所以说是 “potentially applicable”, 是因为它们未必真的需要应用
举个例子: 假设 perl 有一个 replacement, 而 coreutils 依赖 perl, 但这个依赖只发生在 build time, 也就是说 coreutils 在 run time 并不依赖 perl
既然如此, coreutils 的构建结果就可以原样使用, 并不需要再去 graft 它

但问题来了, 我们怎么判断某个依赖是不是纯 build-time dependency 呢
包的 native-inputs 字段 通常会列出 build-time dependencies, 但它终究只是一种提示
从根本上说, 真正的 run-time dependencies, 也就是我们所说的 references, 是 build daemon 的 garbage collector 在 构建结果本身 里扫描出来的那部分依赖
关于 GC 如何扫描 references, 可以直接看 Eelco Dolstra 的博士论文 第 5.5.1 节
回到前面的例子里, 我们必须先真的把 coreutils 构建出来, 才能判断它在 run time 是否依赖 perl

Guix 会尽量只在确实有必要时才去 graft
在上面的例子里, guix build coreutils 返回的结果会和 guix build coreutils --no-grafts 一样
相反, 由于 inkscape 在 run time 确实依赖 perl, guix build inkscape 返回的 derivation 就会把 perl 的 replacement graft 到原始的 inkscape 构建结果上, 也就是 guix build inkscape --no-grafts 返回的那个结果

按照 Mokhov 等人在 Build Systems à la Carte 一文中的说法, grafts 可以看作 dynamic dependencies 的一种形式
换句话说, grafting 是否发生, 取决于中间阶段产生出来的 build result

> Grafts, 用户界面, 与性能

从概念上说, 我们要不要 graft 一个包, 取决于 ungrafted 包的构建结果里到底有哪些 references
但在实际使用里, 我们通常又希望 guix install 这类命令先把整体的 build plan 打出来, 特别是在传了 --dry-run 的时候:

$ guix install inkscape
The following package will be installed:
   inkscape 1.0

71.3 MB will be downloaded:
   /gnu/store/xq64iaxx2gmlcgnipj31wjxlf1yd2g2p-gsl-2.6
   /gnu/store/zrmhnj3pwchn2msphgnwzwd3q89m46rn-aspell-0.60.8
   /gnu/store/5g31zf21lk8nrfd2b8qrm19nwdm9gir9-potrace-1.16
   /gnu/store/qpr7387bsjs7rpl6flrwdvn2zbzh5bch-ghostscript-with-cups-9.52
   /gnu/store/7y3lvk3xf4im8n44337mc6y0ccysvfia-font-dejavu-2.37
   /gnu/store/95n3zvzxzq2bxvdvz292cg04757ij30x-cups-minimal-2.3.1
...

为了兼顾这一点, 1.1.0 之前的 graft 实现采用了这样的策略:
当 substitutes 可用时, 它会直接从 substitutes 那边拿 ungrafted 包的 references 列表
只有在某个 ungrafted 包没有 substitute 可用时, 才会先尝试把那个包实际构建出来
于是, 当 substitutes 存在时, guix install 这类命令就能提前把 build plan 打印出来
但只要某个包没有 substitute, 用户看到的就会是 Guix 一声不响地直接开始构建, 却不先告诉你 build plan 是什么, 这显然 很容易让人困惑

更糟的是, grafting 是按 package 粒度处理的
也就是说, 每次把某个 package lower 成 derivation 时, 你都得重新回答这个问题: “这个 包到底有没有 substitute, 如果有, 它又该不该 graft?”
最后的结果就是 资源使用很差, 用户界面反馈也很糟糕
只要某个包是 graft candidate, 用户就会不断看到那行著名的输出:

updating substitutes from 'https://ci.guix.gnu.org'...

当你用 guix system 去构建整套系统时, 这个问题尤其明显, 因为这种包通常会非常多
更麻烦的是, 这些输出的背后, 往往大致对应着一次新的 TLS 连接和一次 HTTP GET 请求
这不仅慢, 还非常烦人
也许对一部分用户来说, 这种不断抖动的 updating substitutes 提示, 足以说明开发者能力堪忧
而说实话, 对我们开发者中的一些人来说, 为了 grafts 这套机制的精巧设计付出这点代价, 当时似乎也还能忍

对于关闭 substitutes, 选择全本地构建的用户来说, 情况也没好到哪里去
所有可能需要 graft 的包都会一个一个顺序构建, 结果就是完全错过了 --max-jobs 本来能够提供的并行化机会

> 收集 dynamic dependencies

要解决这个问题, 就必须把这些零散的 dynamic dependencies 统一收集起来, 而不是继续一个一个地处理
从概念上说, 我们真正想做的事情大致是这样的:
先做第一遍 lowering, 暂时假装 grafting 是关闭的, 把所有 derivation 都算出来并构建完成
然后再做第二遍, 判断依赖图里哪些 package 真的需要 graft, 再据此计算对应的 grafting derivation
这样一来, 性能问题就好解决了: substitutes 的查询和包的构建都可以尽量并行, 不必再一个个串着跑
如果再把第二遍计算显式暴露给用户界面层, 那用户界面问题也能一起解决, 因为它就能显示两份 build plan: 一份是 “ungrafted” 的, 另一份则是 graft 之后的

问题在于, 我们现有的 API 天生就是串行的
package-derivation 这个函数只接收 一个 package, 对它做 lowering, 然后返回对应的 derivation:

(use-modules (guix)
             (gnu packages base)
             (gnu packages inkscape))

(define s (open-connection))

(package-derivation s coreutils)
⇒ #<derivation /gnu/store/rpfdbax1py483m9ydhvp73s7dgmn6xh4-coreutils-8.31.drv => /gnu/store/jkj7wxybgcpdamkl6fz7wwbb1ak5wxvk-coreutils-8.31-debug /gnu/store/zibwkb5xavnv6z3gzknfqjsxb9b0izh0-coreutils-8.31 7f6c92e3a000>

(package-derivation s coreutils #:graft? #f)
⇒ #<derivation /gnu/store/rpfdbax1py483m9ydhvp73s7dgmn6xh4-coreutils-8.31.drv => /gnu/store/jkj7wxybgcpdamkl6fz7wwbb1ak5wxvk-coreutils-8.31-debug /gnu/store/zibwkb5xavnv6z3gzknfqjsxb9b0izh0-coreutils-8.31 7f6c92e3a000>

(package-derivation s inkscape)
⇒ #<derivation /gnu/store/jzm2zsq8m0rj8wdsmikc0p2ik0cprrcf-inkscape-0.92.4.drv => /gnu/store/clj8rjnsip8a35hyd9nf4l65w7ahn0gs-inkscape-0.92.4 7f6c9c15b730>

(package-derivation s inkscape #:graft? #f)
⇒ #<derivation /gnu/store/psd31x1fq0v2g594z217mh56xzg21dym-inkscape-0.92.4.drv => /gnu/store/zz28ckjwfxwkx3gsm8sc452kmvfiky6y-inkscape-0.92.4 7f6c90ad4f50>

lowering 的过程本身就包含了 graft 的处理逻辑, 这也是为什么我们最后会落到逐个处理带来的低效里
一个直接但代价很高的办法, 是把整套 API 都改成 “复数版本”, 让 package-derivation 及其相关函数都接受 package 列表 而不是单个 package
但这会牵扯到大量改动, 而且最终得到的 API 也不会很好用, 因为从可理解性上讲, 一次处理一个对象始终更自然

1.1.0 里真正采用的解法, 则建立在这样一个观察上: package-derivation 的调用图, 其实和 package graph 本身是镜像关系
因此, 我们可以借助 monad trickery, 也可以借助所谓的 “control effects”, 去收集这些 dynamic dependencies
最后我们选了后者, 因为它没有 monad 那种 “污染整条调用链” 的问题, 写出来的代码也更简单

起点在于, 按定义来说, 带有 dynamic dependencies 的代码最终一定会调用 build-derivations
借助 Guile 里的 delimited continuations, 我们给 build-derivations 加了 instrumentation, 让它在被调用时会 abort 到一个 “build handler” prompt
这个 build handler 会收到两样东西: 一份待构建 derivation 的列表, 以及一个 continuation, 调用它以后就能恢复先前中断的计算并真正开始构建
用户界面层于是就可以装入一个 build handler, 先把接下来要构建的内容展示出来:

(with-build-handler (lambda (continue store things mode)
                      (show-what-to-build store things)
                      (continue #t))
  …)

如果你想实现 dry run, 那么只要不去调用 continue, 就什么都不会真正构建
这里的描述略作了简化, 真正的实现请直接看 build-notifier

接下来我们要做的, 就是利用这套机制把零散的 build-derivations 调用先收集起来, 再在后面统一发出一次真正的 build-derivations
换句话说, 我们想先把所有和 ungrafted 包有关的调用集中到一起, 一次性构建完, 然后再恢复 graft 的计算过程

为此, 我们写了一个 build handler, 它在被调用时不会立刻去构建, 而是返回一个 <unresolved> 对象, 用来把待构建项和 continuation 一起保存下来
与此同时, 我们还提供了一个可以 引入并行性 的 primitive, 使得当某个 dynamic dependency 出现时, 计算不会立刻停住, 而是尽可能继续往下推进, 直到把能算的部分都算完为止
这对应的就是 build-accumulator 和 map/accumulate-builds
其中 map/accumulate-builds 可以理解为一个带有 build 请求收集能力的 map

这个方案和 à la Carte 论文里讨论的 suspending scheduler 有些相似, 只是这里的分解方式没有论文里那么规整
它仍然只是一种近似, 并不是处理 dynamic dependencies 的最优解, 现在也仍然存在 一些会暴露出这个近似性的场景
但总体来看, 它已经是一次显著改进
和 之前尝试过的其他方案 相比, 这个方案有一个明显优点: 它足够正交, 也足够简单
整个改动只有 不到 100 行新增代码, 甚至还顺手从 graft 实现里 删掉了大约 30 行
光是这一点, 就已经很让作者满意了 :-)

> 插曲: 这算是一种 scatter/gather 模式吗

说到底, 我们做的事情无非是把所有 build-derivations 调用先收集起来, 合并成一次调用, 然后再把结果依次送回原来的 continuation
类似的问题其实在远程过程调用, 以及一串 HTTP 请求里也会出现, 所以这里也许藏着一个更一般性的模式
例如, 假设你有这样一段代码:

(map (lambda (thing)
       (http-get (thing->url thing)))
     lst)

如果我们也能把里面所有的 http-get 调用统一捕获起来, 再把它们改写成一串 pipelined GET requests, 最后再用结果恢复对应的 continuation, 那不是很有意思吗

我目前还没找到一个标准的 functional pattern 能很好地描述这件事, 如果你有想法, 非常欢迎交流

> 各种形态的 dynamic dependencies

前面我们已经看到 Guix 是如何处理 dynamic dependencies 的
Nix 也支持一种类似但更受限的 dynamic dependency 形式, 也就是 Nix 语言里的 import primitive
它 可以接收某个 derivation 构建结果, 但并不会尝试把由此产生的 buildPaths 调用统一收集起来

如果再往后退一步看, 会发现 Nix 和 Guix 其实都支持其他形式的 dynamic dependencies
例如, 完全可以写出这样的 derivation: 它的结果依赖于另一个 derivation 构建结果的 reference graph
在 Guix 里, 这是通过给 gexp->derivation 传入 #:references-graphs 实现的, build daemon 随后会 把 reference graph 注入到构建环境里

另一种 dynamic dependency 则是 derivation-building derivations, 也就是 recursive derivations, 这个特性最近已经在 Nix 中实现了
它支持另一种 dynamic dependency 形式: 一个 derivation 的构建过程本身, 可以再去创建并构建新的 derivation
在调度理论里, 这类东西可以看作 moldable tasks
这是一项非常强的能力, 因为它意味着 Nix 不只可以在 package 的层面组合构建过程, 还可以进一步深入到 package build process 的内部, 以更细的粒度来使用

而 Guix 还支持另外一种 dynamic dependency 形式
新的 guix deploy 工具, 底层正是通过 远程求值 g-expressions, 也就是 gexps 来工作的
例如, 在真正部署一个 operating system 之前, 它会先在远端节点上执行一段代码, 做一些 sanity checks:
比如检查所声明的 file system UUID 或 label 是否有效, 检查是否需要把额外的 kernel module 加入 initrd, 等等
为此, remote-eval 会先构建出一个产生 Scheme 程序的 derivation, 再把它连同全部依赖一起部署到目标机器上, 执行之后再把结果取回来
这种 dynamic dependency 形式, 同样也能从前面提到的收集机制中获益

> 结语

这篇文章围绕 Guix 设计与实现中的一个细节讲了很久, 但这个话题确实值得展开
grafts 对 functional deployment 在生产环境中的使用至关重要, 因为它直接关系到 security updates 能不能及时交付
而如果这套机制会明显拖累用户体验, 那显然就很难让人满意

1.1.0 之前的 graft 实现, 由于按 package 粒度顺序处理, 对用户界面和性能都造成了明显负面影响
在 1.1.0 里, 我们借助 delimited continuations 把 grafts 这类 dynamic dependencies 收集起来, 批量完成构建, 然后再恢复每个 derivation 的计算

结果也很清楚: dynamic dependencies 的实现, 其实牵出了大量有意思的设计与实现问题
而这件事大概也还远远没有走到终点
